--------------------------------------------------------------------------------
1. PROJECT OVERVIEW
--------------------------------------------------------------------------------

Project Name    : Secure File Transfer / Secure Data Transfer
Purpose         : Encrypt files or text on the server, store in database as
                  .encrypted format, and allow secure retrieval using Access
                  Code and optional Decryption PIN. Supports NFC sharing and
                  limited downloads.

Key Features    :
  - Encrypt files or paste text before upload
  - Store only encrypted data in MongoDB (never plaintext)
  - Two download modes: encrypted file (Access Code only) or decrypted file
    (Access Code + PIN)
  - Shareable download link for decrypted file
  - NFC sharing to transfer link to another device without internet
  - Configurable max download count (1 to N)
  - 24-hour automatic expiry (TTL)
  - Max file size: 150 MB


--------------------------------------------------------------------------------
2. TYPES OF ENCRYPTION AND CRYPTOGRAPHIC METHODS USED
--------------------------------------------------------------------------------

2.1 SYMMETRIC ENCRYPTION - FERNET
---------------------------------
  - Algorithm       : Fernet (from Python 'cryptography' library)
  - Type            : Symmetric (same key for encrypt and decrypt)
  - Underlying      : AES-128 in CBC mode with HMAC-SHA256 authentication
  - Key size        : 128-bit AES key + 128-bit signing key (256 bits total)
  - Why Fernet      : Authenticated encryption; prevents tampering; easy API

  - Usage in project:
    * Plaintext (file bytes) is encrypted with a derived key before storing.
    * Only encrypted_data is stored in MongoDB.
    * Decryption happens only on the web server when user provides valid PIN.


2.2 KEY DERIVATION - PBKDF2-HMAC
---------------------------------
  - Algorithm       : PBKDF2 (Password-Based Key Derivation Function 2)
  - Hash            : HMAC with SHA-256
  - Purpose         : Convert 4-digit PIN into a proper encryption key for Fernet
  - Parameters      :
      - Salt        : Static value (b"unique_salt_value") - per-record salt
                      recommended for stronger security
      - Iterations  : 100,000 (makes brute-force slower)
      - Key length  : 32 bytes (256 bits)
      - Output      : Base64url-encoded for Fernet compatibility

  - Why PBKDF2      : PIN is short; PBKDF2 stretches it into a strong key and
                      slows down brute-force attacks.


2.3 HASHING - SHA-256
---------------------
  - Algorithm       : SHA-256 (Secure Hash Algorithm 256-bit)
  - Purpose         : Store a hash of the decryption PIN in the database instead
                      of the PIN itself. Used to verify PIN on download without
                      storing plaintext PIN.
  - Usage           : key_hash = SHA256(PIN) stored in DB; on download,
                      hash(user_input) is compared with key_hash.

3. METHODS AND TECHNIQUES USED
--------------------------------------------------------------------------------

3.1 SERVER-SIDE ENCRYPTION
  - Encryption and decryption are performed only on the server (web application).
  - Files are never stored in plaintext; only .encrypted (ciphertext) is in DB.
  - Flow: Upload -> Server encrypts -> Store ciphertext -> Return Access Code + PIN.

3.2 TWO-FACTOR ACCESS
  - Access Code : Unique 4-digit (or 6-digit on collision) code to identify
                  the transfer. Required for any download.
  - Decryption PIN : 4-digit PIN. Optional for download:
      - With PIN    : Server decrypts and returns original file.
      - Without PIN: Server returns encrypted file (.encrypted) as-is.

3.3 ATOMIC DOWNLOAD COUNT
  - MongoDB update_one with $inc is used to decrement downloads_remaining.
  - Condition: only update when downloads_remaining > 0.
  - Prevents race conditions when multiple users download simultaneously.

3.4 AUTO-EXPIRY
  - MongoDB TTL index on "created_at" with expireAfterSeconds = 86400 (24 hours).
  - Additional cleanup: cleanup_expired_records() on each upload (delete older than 24h).

3.5 SHAREABLE LINK
  - Format: https://<domain>/download?code=<access_code>&key=<decryption_pin>
  - GET /download with query params returns a small HTML page with an
    auto-submitting form that POSTs to /download to trigger file download.
  - This avoids Vercel serverless response size limits and ensures download
    works when link is opened from NFC or another device.

3.6 NFC SHARING
  - Web NFC API (NDEFReader) used to write the download URL to an NFC tag.
  - User taps phone to NFC tag; recipient taps same tag to read URL.
  - Transfer of link happens without internet; download happens when URL is opened.

4. TECHNOLOGY STACK
--------------------------------------------------------------------------------

  - Database   : MongoDB (e.g. Atlas), PyMongo 4.9.1 with SRV support
  - Crypto     : cryptography 41.0.3 (Fernet, PBKDF2HMAC, SHA256)

--------------------------------------------------------------------------------
5. DATABASE MODEL (MongoDB collection: transfers)
--------------------------------------------------------------------------------

  access_code         : string, unique index (identifies the transfer)
  key_hash            : string, SHA-256 hex digest of decryption PIN
  filename            : string, original filename (e.g. document.pdf)
  encrypted_filename  : string, filename + ".encrypted" for encrypted downloads
  size_mb             : float, approximate size in MB
  encrypted_data      : binary (BSON Binary), Fernet ciphertext only
  created_at          : datetime, used for TTL index (24h expiry)
  downloads_remaining : integer, decremented on each successful download
  content_type        : "file" | "text"

  Indexes:
  - (access_code, ASCENDING), unique
  - (created_at, expireAfterSeconds: 86400)  // TTL 24 hours


--------------------------------------------------------------------------------
6. API ROUTES AND BEHAVIOUR
--------------------------------------------------------------------------------

  GET  /              : Renders main page (upload form + download forms).
  POST /upload        : Accepts file or text; encrypts; stores in DB; returns
                        JSON with code, key, filename, size_mb, download_limit.

  GET  /download      : Query params: code, key (optional).
                        Returns HTML page with auto-submit form that POSTs to
                        /download so that file download is triggered via POST.

  POST /download      : Form params: code, key (optional).
                        If key empty: return encrypted file (.encrypted).
                        If key present: verify key_hash, decrypt, return original
                        file. Uses atomic $inc to decrement downloads_remaining.
                        Returns 413 if payload too large; error page on invalid
                        code/PIN/expired/limit reached.


--------------------------------------------------------------------------------
7. SECURITY DESIGN SUMMARY
--------------------------------------------------------------------------------

  - Only encrypted data is stored; plaintext never persisted.
  - PIN is never stored; only key_hash (SHA-256) is stored.
  - Access Code + PIN (for decryption) both required for decrypted download.
  - File size limit (150 MB) enforced on upload and in Flask config.
  - TTL and download limit reduce exposure and storage.
  - Encryption/decryption only on server; DB holds only .encrypted format.

8. SOLUTIONS TO IMPLEMENTATION CHALLENGES
--------------------------------------------------------------------------------

  8.1 Download count not working / only once
      - Cause: max_downloads was not sent from frontend (input had no "name"),
               and duplicate decrement logic for text vs file.
      - Solution: Added name="max_downloads" and appended to FormData in JS;
               single atomic update_one with $inc for both encrypted and
               decrypted download; no duplicate decrement.

  8.2 Encrypted download error (404)
      - Cause: Form pointed to /download_encrypted which was removed.
      - Solution: Form action changed to /download; when key is empty, server
               returns encrypted file.

  8.3 File not downloading when clicking link (e.g. from NFC)
      - Cause: GET /download returning binary file can fail on Vercel
               (response size/streaming) or browser not triggering download.
      - Solution: GET /download returns small HTML page with form; form
               auto-submits via POST to /download; POST returns file with
               mimetype application/octet-stream and Content-Disposition
               attachment. Added download_redirect.html template.

  8.4 Vercel 500 / FUNCTION_INVOCATION_FAILED
      - Cause: Possible ReturnDocument import or find_one_and_update issues
               in serverless; index creation failing.
      - Solution: Removed pymongo.return_document; use update_one with $inc
               instead of find_one_and_update; wrapped index creation in
               try/except; increased MongoDB connection timeout.

  8.5 NFC share showing multiple options instead of only NFC
      - Solution: Use only Web NFC (NDEFReader.write) for the NFC button;
               removed Web Share API from that flow so system NFC only is used.

9. POSSIBLE VIVA QUESTIONS AND SHORT ANSWERS
--------------------------------------------------------------------------------

  Q: What type of encryption is used?
  A: Symmetric encryption using Fernet (AES-128-CBC with HMAC-SHA256 for
     authentication). Key is derived from a 4-digit PIN using PBKDF2-HMAC-SHA256.

  Q: Why not store the PIN in the database?
  A: For security. Only the hash of the PIN (key_hash) is stored. We verify
     the user’s PIN by hashing it and comparing with key_hash. Plaintext PIN
     is never stored.

  Q: What is key derivation and why is it used?
  A: Key derivation (PBKDF2) turns a short PIN into a full encryption key.
     It uses many iterations (100,000) to make brute-force attacks slower and
     produces a 256-bit key suitable for Fernet.

  Q: Where is the file encrypted and decrypted?
  A: Only on the server (web application). The browser sends plaintext; the
     server encrypts before storing. On download, the server decrypts and
     sends the file. The database always holds only encrypted data.

  Q: How is download limit enforced?
  A: We store downloads_remaining in MongoDB. On each successful download we
     use an atomic update: $inc to decrement by 1, only when
     downloads_remaining > 0. When it reaches 0, the record is deleted.

  Q: How does the shareable link work?
  A: The link contains code and key in the query string. Opening it (GET)
     loads a small HTML page with a form that auto-submits via POST to
     /download. The POST response is the file, so the browser starts the
     download. This avoids issues with large GET responses on serverless.

  Q: What is NFC used for in this project?
  A: To share the download URL to another device without internet. We use
     the Web NFC API to write the URL to an NFC tag; the recipient taps
     the tag to get the link and opens it when online to download.

  Q: What is the role of MongoDB TTL?
  A: TTL (Time To Live) index on created_at with 24-hour expiry automatically
     deletes old documents, so files do not stay in the database forever.

--------------------------------------------------------------------------------
11. SUMMARY FOR VIVA
--------------------------------------------------------------------------------

  This project implements a secure file transfer system where:
  (1) Files/text are encrypted on the server using Fernet (symmetric) with a
      key derived from a PIN via PBKDF2-HMAC-SHA256.
  (2) Only encrypted data is stored in MongoDB; the PIN is stored only as
      its SHA-256 hash.
  (3) Users can download either the encrypted file (with Access Code only) or
      the decrypted file (with Access Code + PIN).
  (4) A shareable link and NFC sharing allow transferring the download
      capability to another device; download is triggered via an auto-submit
      POST to avoid serverless limits.
  (5) Download count is enforced with atomic MongoDB updates, and records
      expire automatically after 24 hours (TTL).
